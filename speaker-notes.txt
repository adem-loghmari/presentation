SLIDE 4 — CONTEXT: why generic e-commerce breaks
-------------------------------------------------------------------------------
"Live events have two properties ordinary e-commerce ignores: they are SPATIAL
and they are REAL-TIME. A product maps to an order; a seat does not. That gap
creates four concrete pains for organizers:"

 - Fragmented tools — they use one app to design the layout, another to sell, a
   third to scan at the door. Nothing talks to each other.
 - No spatial rendering — standard platforms can't draw a custom seating chart;
   at best they overlay a static image, which can't carry per-seat pricing.
 - Race conditions — the key metric here is the latency of the seat lock: the gap
   between a user picking a seat and that seat becoming unavailable to everyone
   else. If that window is wide, two people buy the same seat. For commercial
   ticketing that's a catastrophic failure.
 - Walled gardens — generic platforms charge high commissions and expose no
   embeddable API, so organizers can't sell tickets on their own websites.

"getseat exists to close all four."

SLIDE 7 — TECHNOLOGICAL STACK
-------------------------------------------------------------------------------
"The stack is modern and deliberately decoupled."

 - Next.js 16 / React 19 — hybrid rendering: server-side for SEO-critical event
   pages, client-side for the interactive canvas. UI on Material UI.
 - Konva.js — standard DOM can't efficiently render or hit-test thousands of
   shapes; Konva gives a WebGL-accelerated 2D canvas with built-in transformer,
   hit detection, and layer separation. This is what makes 60 FPS possible.
 - Laravel 12 / PHP 8.3 — chosen for Eloquent ORM and, crucially, first-class
   database transactions and row locking — exactly what ACID-compliant ticketing
   needs.
 - Reverb / Pusher — WebSocket broadcasting. Self-hosted Reverb in development,
   Pusher in production for guaranteed availability. Laravel Echo on the client.
 - SkipCash — a regional gateway (QAR / GCC currencies) with HMAC-SHA256 signed
   webhooks for cryptographic verification.
 - Spatie Permission — role-based access control on top of Laravel Auth.

Don't read every line if short on time — name the front three and the back three.


SLIDE 8 — ARCHITECTURE (headless)
-------------------------------------------------------------------------------
"The architecture is headless — a decoupled model. The Next.js frontend is pure
presentation; all logic, auth, and state live in the Laravel API. They talk over
REST plus WebSockets, and MySQL sits behind with ACID transactions and
pessimistic locks."

Why it matters:
 - Frontend and API scale independently.
 - WebSockets replace HTTP polling — and that real-time channel is precisely
   what eliminates double-booking in the live UI.
 - One business-logic layer serves three consumers at once: the web client,
   embedded iframe widgets, and the partner SDK. We never duplicate logic.

If asked monolithic vs decoupled: "A monolith server-renders views and polls; the
decoupled model gives an interactive Konva canvas, real-time sync, and a reusable
JSON API — at the cost of a distributed deployment, which the CD pipeline handles."

SLIDE 10 — DATA MODEL (hierarchy)
-------------------------------------------------------------------------------
"Ticketing needs spatial and temporal awareness, so the schema enforces a strict
four-level hierarchy:"

 - EVENT — the root: title, timezone-aware UTC dates, an owner, encrypted payment
   credentials, and a many-to-many staff relationship via a pivot table.
 - VENUE TEMPLATE → VENUE — the template stores the pristine master layout as a
   JSON blob; a Venue is an event-specific deployment of that template.
 - TIMESLOT — a venue can run multiple sessions (a 2pm and an 8pm show). Crucially,
   it's the TimeSlot — not the Venue — that owns the Location records. That's how
   availability is isolated per session.
 - LOCATION — the lowest unit: an individual bookable seat, table, or standing
   area. It carries the shape_id from the canvas, a type, a capacity, and a parent.

"On the transactional side there's a parallel chain: Reservation → Order →
Transaction → Ticket → ScanLog — the full financial lifecycle from a temporary
hold all the way to a scan at the gate."


SLIDE 11 — ACCESS CONTROL (RBAC)
-------------------------------------------------------------------------------
"Authorization works on two layers."

 - Platform roles (Spatie): user (a normal organizer), support (read-only admin
   plus refunds), and super-admin (full authority — ban users, delete events,
   monitoring).
 - Event-scoped roles (pivot table): within one event, a collaborator is a viewer
   (dashboards, manual check-ins, door scanning — read only) or an editor (all of
   that plus venue write access). Both are enforced server-side by Laravel Policies.
 - Magic-link invitations: an organizer invites by email; if the user exists they're
   linked instantly, otherwise a 64-character token with a 7-day expiry is emailed.
 - Venue deletion is a deliberate two-step workflow: because a venue can have live
   tickets, an editor can only REQUEST deletion; the owner must APPROVE. Nothing
   with live reservations is destroyed without explicit consent.

Pick two of the four if time is tight — the viewer/editor split and the deletion
workflow are the strongest.

SLIDE 13 — VENUE BUILDER · PERFORMANCE
-------------------------------------------------------------------------------
"A stadium-scale map can have over 5,000 interactive nodes. If you let React
re-render that naïvely, the browser freezes. Three disciplines keep it at 60 FPS:"

 - Layer separation — the canvas is split into independent Konva layers: a static
   grid that never re-renders, the interactive shape layer (the only one that
   updates on edits), and a preview layer for ephemeral drawing feedback so a
   mouse-move never re-renders real shapes.
 - Viewport culling — useDerivedVenueState computes the visible bounding box from
   the stage's x, y, and scale; only shapes inside it are mounted. Everything off-
   screen carries zero render and zero hit-test cost. There's also a hard cap of
   4,000 rendered shapes per zone as a safety valve.
 - Memoization — every shape component is wrapped in React.memo, the rendered list
   is useMemo'd, and handlers are stabilized with useCallback.

"All of this is orchestrated by 16 specialized hooks — each owns exactly one
concern: state, drawing, drag-and-snap, transformation, navigation, sync, and so
on. That separation is what kept the most complex module maintainable."

If asked about rotation math: "When you rotate a group, each shape's absolute x/y
is recomputed with a 2D rotation matrix around the group's pivot — otherwise the
hitbox drifts away from where the seat is drawn."


SLIDE 14 — VENUE BUILDER · SMART TOOLS
-------------------------------------------------------------------------------
"On top of the canvas are tools that remove the manual tedium:"

 - Magic-Wand auto-labeling — manually numbering thousands of seats is impossible.
   The tool sorts selected seats by Y into rows (with a one-seat-diameter
   tolerance), sorts each row left-to-right by X, then assigns Row A, B, C… and

 - AI text-to-map — a non-technical organizer types, say, "a wedding layout with a
   center stage, 10 round tables, and a VIP standing zone for 200." That prompt
   goes to an LLM constrained to a strict JSON schema, which returns valid shape
   objects with computed coordinates — injected straight into the canvas.
 - Minimap + grid snapping — a draggable viewport rectangle for navigating large
   maps, and a 10-pixel grid snap so adjacent rows align perfectly with no manual
   pixel-pushing.
 - Multi-floor + blueprints — floor tabs and zone drill-down for theatres and
   arenas; and you can upload an architectural blueprint, trace your seats over it
   as a semi-transparent background, then fade it to zero.

Lead with Magic-Wand and AI text-to-map — those land best.


SLIDE 15 — CANVAS → DATABASE (serialization)
-------------------------------------------------------------------------------
"Now the bridge from a visual editor to strict storage."

 - The entire canvas serializes into one canvas_data JSON object — floors,
   categories, shapes. It's deliberately flat and denormalized because during the
   design phase it's cheap to write and read as a single blob.
 - On publish, the backend's syncLocations() migrates it in three passes:
     1) categories  → PriceTier rows (updateOrCreate by ui_category_id),
     2) bookable shapes → Location rows (keyed by shape_id),
     3) a LocationPrice link between each location and its tier, then parent-child
        relationships are resolved.
 - Because it keys on stable IDs, anything removed from the canvas — orphaned tiers
   or locations — is pruned automatically. The JSON and the relational tables stay
   in sync.

"So the design phase is fast and fluid; the publish step is atomic and strict.
This same method is reused when a B2B partner injects pricing via the SDK."


SLIDE 16 — RESERVATION ENGINE (pessimistic locking)
-------------------------------------------------------------------------------
"This directly answers the 'never double-book' question. When a guest locks a seat:"

 1) A database transaction begins.
 2) The specific Location row is fetched with lockForUpdate() — that tells MySQL
    to block every other read and write on that row until this transaction commits.
 3) The system counts active holds + sold tickets. If held + sold + requested >
    capacity, it returns 409 Conflict immediately.
 4) Otherwise it creates a Reservation with a 10-minute expiry tied to the user's
    session token, and commits.
 5) A second request that arrived mid-transaction is forced to WAIT; when it's
    unblocked it sees the updated count and gets a 409. There is no window for a
    race — the database itself serializes contention on that row.

Add: "Expired reservations are pruned automatically by the Prunable trait and
Laravel's scheduler. And availability is computed with aggregate counts —
sold, locked-by-others, locked-by-me, and remaining — which is what keeps that
endpoint under 100 ms even on large venues."

If asked about idempotency: "For single-capacity seats, if you already hold it
under your session token, the lock is skipped — so slow network retries don't
create duplicate holds."


SLIDE 17 — REAL-TIME SYNC (WebSockets)
-------------------------------------------------------------------------------
"Database locks prevent double-booking at the data layer, but other users would
see stale seats until they refresh. So we push state instantly with three
broadcast events:"

 - SeatLocked — a seat was just reserved.
 - SeatUnlocked — a hold was cancelled or expired.
 - SeatSold — a payment was confirmed.

Key points:
 - Each broadcasts on a per-timeslot channel (venue.{id}.timeslot.{id}) so the
   evening show's events never bleed into the morning show's map.
 - They implement ShouldBroadcastNow — they bypass the queue for immediate delivery.
 - On the client, Laravel Echo + Pusher flip a seat from available to locked within
   milliseconds on every connected screen.
 - The client cart is a state machine running the 10-minute countdown; when it
   expires, seats are ejected from the UI and reverted. And a "Best Available"
   algorithm auto-selects the best contiguous block of N seats — grouping by row,
   preferring front rows for visibility.

"This is the moment to demo two browsers side by side if possible."


SLIDE 18 — PAYMENTS (never trust the browser)
-------------------------------------------------------------------------------
"Payments are built on one principle: never trust the browser to confirm money."

The four-step lifecycle:
 - INITIATE — decrypt the event's stored payment secrets, build the SkipCash
   payload, sign it with HMAC-SHA256, POST to the gateway, create a Transaction
   record, and return the payment URL.
 - REDIRECT — the guest pays on SkipCash's hosted page.
 - VERIFY — when they return, the SERVER polls SkipCash directly, server-to-server,
   for the real status. The frontend is never trusted to self-report success.
 - FULFILL — tickets are issued only after that cryptographic confirmation.

Add:
 - Payment secrets are encrypted at rest with Laravel's Crypt.
 - The layer is pluggable: a TransactionStatus enum normalizes each gateway's
   vocabulary ("paid" → SUCCESS, "canceled" → FAILED…), so adding Stripe later
   touches no controller code.
 - Free events use a native, gateway-free checkout — the order completes
   immediately with no external redirect.


SLIDE 19 — TICKETS & CHECK-IN
-------------------------------------------------------------------------------
"After payment, the closing sequence:"

 - ISSUE — every reservation converts to a Ticket with a unique UUID barcode_hash;
   the holds are deleted to free the lock, and a SeatSold event is broadcast.
 - DELIVER — a confirmation email is queued, and there's a server-rendered PDF
   ticket download (via dompdf) showing event, venue, time slot, seat, and barcode.
 - SCAN — at the door, a browser-based QR scanner uses the device camera — no
   proprietary hardware. The backend validates: invalid → 404, already scanned →
   400, valid → marks it scanned and writes a ScanLog. It also checks the ticket
   belongs to this event, preventing cross-event fraud.
 - ANALYZE — organizers get a live dashboard: total capacity, tickets sold, total
   revenue, a 7-day sales-velocity chart, and ticket distribution by price tier.

"So the platform automates the entire flow, from seat selection to venue entry."


SLIDE 20 — SECURITY HARDENING
-------------------------------------------------------------------------------
"Turning a prototype into a production SaaS meant hardening the auth layer."

 - The biggest change: we moved from Sanctum tokens in localStorage to HTTP-only,
   Secure, SameSite cookies. localStorage is readable by any injected JavaScript —
   an XSS bug could exfiltrate the token. An HTTP-only cookie is physically
   invisible to JavaScript, so that theft vector is eliminated. For a platform
   handling payments, that was non-negotiable.
 - CSRF: the API client calls GET /sanctum/csrf-cookie before every mutating
   request, seeding the XSRF token Laravel validates — the standard SPA pattern.
 - Google OAuth 2.0 via Socialite for frictionless sign-in.
 - Embed tokens: inside an iframe, SameSite policies block the cookie session, so
   embedded requests use a scoped Bearer token instead — and that path only
   activates for /embed/ routes, keeping the security boundary clean.


SLIDE 21 — B2B DEVELOPER SDK
-------------------------------------------------------------------------------
"This is what turns getseat from an app into a white-label engine."

 - API keys are scoped to a partner:api ability; the plain token is shown once and
   never stored — every SDK endpoint checks tokenCan('partner:api').
 - Five server-to-server endpoints under /v1/embed/ — create a session, publish a
   venue from a template, confirm orders, and inject pricing categories.
 - The clever part is shadow users. A partner's customer has no getseat account.
   So the partner calls /v1/embed/session with their own external_user_id; the
   backend firstOrCreates a synthetic User keyed by (partner_id, external_user_id),
   with a generated email and random password, and issues a 2-hour scoped embed
   token. That user can lock seats, check out, and get tickets — completely proxied
   through a managed identity, never forced to register.
 - Then the partner drops an iframe — /embed/reserve for the portal, /embed/builder
   for white-label map editing — with a stripped layout and strict CORS.

"publishPartnerVenue lets a partner provision an entire event, venue, and all its
time slots from their own backend in one call."


SLIDE 22 — DEVOPS AUTOMATION
-------------------------------------------------------------------------------
"Deploying a decoupled architecture by hand is risky, so it's fully automated.
Every push to the dev branch triggers a GitHub Actions pipeline:"

 1) INIT — install PHP 8.3 with extensions and Node 22, inject secrets (Pusher,
    SkipCash, DB) from GitHub Secrets, composer install optimized for production.
 2) BUILD — npm ci for a reproducible install, then the Next.js production build.
 3) TRANSFER — an SSH key authenticates to the DirectAdmin server; rsync --delete
    pushes the Next.js out/ to the public web root and the Laravel backend/ to a
    NON-public directory so source and .env are never web-accessible.
 4) EXECUTE — on the server: restore permissions, clear and rebuild route/config/
    view caches, and run migrate --force.

"The whole thing, push to live, takes under four minutes with zero manual steps.
Alongside it there's a three-tier admin panel — users, events, venues — and a
real-time notification engine that keeps organizers and staff in sync on things
like venue-deletion requests and roster changes."

LIKELY QUESTIONS & QUICK ANSWERS
-------------------------------------------------------------------------------
Q: Pessimistic vs optimistic locking — why pessimistic?
A: Ticket drops are high-contention on the SAME seat. Optimistic locking would
   retry-storm under that load; pessimistic locking serializes contention at the
   row and gives a clean, immediate 409 — simpler and correct for this workload.

Q: What if the WebSocket connection drops?
A: The DB lock is the source of truth, so correctness never depends on the socket.
   The portal can re-fetch availability; the socket is an optimization for instant
   UI, not for safety.

Q: Why store the canvas as one JSON blob instead of normalized rows from the start?
A: During design the layout changes constantly — writing one blob is far cheaper
   than thousands of row updates. Normalization happens once, atomically, at publish
   via syncLocations(), which is when strict relational integrity actually matters.

Q: How do you guarantee a paid order isn't faked?
A: The browser never reports payment status. We poll the gateway server-to-server
   and only issue tickets on the provider's confirmation; secrets are encrypted at
   rest and webhooks are HMAC-signed.

Q: How are shadow users prevented from leaking across partners?
A: They're keyed by a composite (partner_id, external_user_id), the embed token is
   scoped and short-lived (2h), and ownership-chain checks verify a template/venue
   belongs to the calling partner before any publish.

Q: Did you measure the KPIs or estimate them?
A: Measured under the validation methodology — concurrency was verified with
   simulated simultaneous bookings (zero double-booking), broadcast latency and API
   response were timed, and 60 FPS was confirmed while interacting with 1,000+ shapes.
