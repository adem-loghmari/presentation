------------------------------- ACT I · THE PROBLEM ---------------------------
[4] DIVIDER. "First — why does this need a new platform?"

[5] CONTEXT. "Ordinary e-commerce: a product maps to an order. A seat doesn't — it
has a position and can be grabbed by two people at once." Four pains: fragmented
tools, no spatial rendering, double-booking (a wide seat-lock window = catastrophe),
walled gardens (commissions, no embeddable API).

[6] PROBLEMATIC. Read slowly: "How do you let an organizer draw a venue once, and
let the world book it in real time — without ever double-booking a seat?
Everything that follows answers this one sentence." This is the spine.

------------------------------ ACT II · THE BLUEPRINT -------------------------
[7] DIVIDER. "Here's the shape of the answer."

[8] STACK. "Modern and deliberately decoupled." Front: Next.js 16/React 19, Konva.js
(why we hit 60 FPS). Back: Laravel 12/PHP 8.3 (chosen for ACID + row locking),
Pusher/Reverb, SkipCash, Spatie. Name the front three and back three; don't read all.

[9] ARCHITECTURE — table. "Headless: the browser draws, the server rules." Walk the
rows; the State-sync row (polling vs WebSockets) is the one that kills double-booking.

[10] DATA MODEL. Event → VenueTemplate/Venue → TimeSlot → Location. Key line: "the
TimeSlot, not the Venue, owns the Locations — that's how availability is isolated
per session." Transactional chain: Reservation → Order → Transaction → Ticket → ScanLog.

[11] RBAC. Platform roles (user/support/super-admin) + event roles (viewer/editor)
via Policies. Magic-link invites; two-step venue-deletion (editor requests, owner
approves).

[13] CREATE EVENT (1/11). "It starts with an Event — the root entity: title, UTC

[14] BUILD THE VENUE (2/11). "The technical nucleus — a Konva.js canvas at 60 FPS

[15] SCHEDULE SESSIONS (3/11). "A venue hosts multiple TimeSlots — a 2pm and an 8pm

[16] PUBLISH → SYNC (4/11). "Going public runs syncLocations() — it migrates the

[17] PAYMENT SETUP (5/11). "The organizer plugs in SkipCash; keys are stored

checkout with no redirect."

[18] GUEST PORTAL (6/11). "Now the guest. The portal loads a live map — sold,
locked, mine, remaining — in one snapshot under 100 ms. They pick a session, or hit
Best-Available to auto-pick the best contiguous block."

[19] SEAT LOCKING (7/11) ★ THE MONEY MOMENT. Slow down here. The clip shows a guest
picking a seat: it's selected, the cart fills, and a "Releasing in 09:55" hold
countdown starts. "This answers the whole problematic. The instant a seat is picked,
lockForUpdate() inside an ACID transaction makes every rival wait, then return 409 —
so the database itself prevents double-booking. The hold lasts 10 minutes — you can
see the countdown — and a SeatLocked event broadcasts over Pusher to flip that seat
on every other connected screen in under 200 ms." (The clip is single-window; if you
want the literal two-screens-syncing shot, capture it on your own machine where the
Pusher realtime channel is confirmed live — see the note at the end.)

[20] PAYMENT (8/11). "Checkout redirects to SkipCash with an HMAC-signed payload.
On return, the SERVER polls SkipCash for the real status — the browser is never
trusted to confirm a payment. Tickets issue only on confirmation."

[21] TICKET (9/11). "Each ticket gets a unique UUID barcode_hash, delivered as a
queued HTML email and a downloadable PDF carrying the QR."

[22] QR SCAN (10/11). "At the door, a browser-based QR scanner — no hardware. It
rejects invalid or already-scanned tickets and writes a ScanLog. Watch: valid, then
the same code again — already scanned."

[23] DASHBOARD (11/11). "And the organizer sees it all live — capacity, tickets
sold, revenue, a 7-day sales-velocity chart, and distribution by price tier. That's
one event, end to end."

DEMO / CLIP TIPS
-------------------------------------------------------------------------------
- Clips auto-play muted and loop; you never click 'play'. If a clip is missing the
  slide shows a labeled placeholder, so rehearse anytime.
- The ONLY clip that must be perfect is slide 19 (two windows, real-time locking) —
  it's the proof of your thesis. Record it cleanly: lock on the left, watch the
  right grey out. If you can also show the 409 on a second attempt, even better.
- Keep clips 2-5 s and end near where they start so the loop is seamless.
- If the room laptop struggles, clips on inactive slides are paused automatically;
  only the visible slide's clip runs.

LIKELY QUESTIONS
- Pessimistic vs optimistic locking? High contention on the SAME row — optimistic
  would retry-storm; pessimistic serializes it and returns an immediate clean 409.
- WebSocket drops? The DB lock is the source of truth; correctness never depends on
  the socket — the portal just re-fetches availability.
- Why one JSON blob then migrate? Design changes constantly; one blob is cheap,
  normalization happens once at publish via syncLocations().
- Faked "paid" order? Browser never reports status; we poll the gateway
  server-to-server; secrets encrypted, webhooks HMAC-signed.
- Shadow-user isolation? Composite (partner_id, external_user_id); scoped 2-hour
  token; ownership-chain checks before any publish.
- KPIs measured? Yes — simulated concurrent bookings (zero double-booking), timed
  latency/API, 60 FPS confirmed with 1,000+ shapes.
